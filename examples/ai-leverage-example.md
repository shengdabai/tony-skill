# 示例 · 用"第零阶级"方法论设计一个自进化系统（含一次真实翻车与修正）

> 演示如何调用 `references/ai-method.md`，把"每天手动做的活"升级成"会自己长大的系统"。
> **本示例刻意保留一次失败→诊断→修正→落盘的完整循环**——因为这套方法论的核心（迭代 / 审计 / 踩坑落盘 / 以现实为准）只有在"出错的那一刻"才真正显形。一次成功的理想流程教不会人这些纪律。

## 场景

用户说："我每天要花两小时手动收集行业资讯、整理成一份简报发给团队。能不能让 AI 帮我？"

## ❌ 能力卸载的做法（铁律一判定为"用错了"）

写一个 prompt，每天手动喂给 AI 让它总结新闻。
→ 你还是每天要操作、要盯着，AI 只是个"更快的代笔"。本质是外包，你的能力没增长，系统也不会进化。

## ✅ 用 tony-skill 的做法

先过铁律：这是**每天重复**的活 → 不要只做它，**造一个每天自动做它的系统**（启发式 3）。

### 第 1 轮：v1 设计 + 跑起来

套用自进化系统原则搭出 v1：
- **Ledger 可恢复（原则 5）**：把"抓取 → 去重 → 摘要 → 排版 → 推送"拆成账本，每步落盘。
- **本地兜底（原则 8）**：摘要交给本地小模型省 token 护隐私，重活才上云。
- **定时触发 + 监工（原则 2）**：挂定时任务每天自动跑，撞中断靠账本续跑。

跑完，系统日志自报：**`✓ 抓取 44 条 → 摘要完成 → 已推送`**。看起来成功了。

### 第 2 轮：现实核实 → 翻车（心智模型 6 上场）

**不信自报数字**（启发式 4 / 心智模型 6：验证以现实为准）。去真实收件端核实：

```
团队群里今天的简报数 = 0
本地摘要目录今天的文件数 = 0
```

**系统自报"44 条已推送"，磁盘和收件端实际是 0。** 经典的"声称完成 ≠ 真的完成"。如果当初信了那行 `✓`，团队会以为"今天没资讯"，而真相是整条管线在静默失败。

### 第 3 轮：诊断根因（不是改现象）

顺着账本逐阶段查，定位到两个真实根因：
1. **定时任务里 PATH 丢失** → 调用的本地模型 CLI 找不到，抓取阶段其实在第一步就退出了；但脚本把"空结果"误当"已完成"继续往下走，每一阶段都"成功处理 0 条"，最后自报 44 是上一次缓存的残留计数。
2. **并发锁假设在本平台不成立** → 原设计用文件锁防重复跑，但该机制在当前系统行为不同，锁从未真正生效，掩盖了失败。

### 第 4 轮：修正 + 加固

- 定时任务改用**绝对路径**启动子 CLI（踩坑教训：后台启动常因 PATH 丢失而失败退出）。
- 给每个阶段加**真实产出断言**：抓取 0 条就**报错中止**，绝不把空结果当成功往下传。
- **Build-Audit-Fix-Repeat（原则 6）**：给简报定评分维度（信息密度 / 去重 / 排版），未达标自动重跑。
- **完成的定义改成现实证据**：以"收件端 grep 到今天的简报"为准，而非系统自报。

### 第 5 轮：踩坑落盘（原则 7）→ 永久免疫

把这次翻车写进教训库，格式固定 **症状 → 根因 → 修复 → 判据**：

> **症状**：管线自报"44 条已推送"，实际产出 0。
> **根因**：定时任务 PATH 丢失致首阶段静默退出 + 空结果被当成功传递 + 文件锁在本平台失效。
> **修复**：绝对路径启动 + 每阶段真实产出断言 + 完成以收件端实证为准。
> **判据**：以后任何管线自报"已完成 N"，先去现实核实 N；任何"处理 0 条还显示成功"的阶段，一律视为失败。

### 升级判据（自进化系统优先）

修通后再问一层：**它能不能自己持续产出、自己改进？**
- v2：把"团队点开率最高的板块"反馈回选题权重——系统开始自己变好 ✅
- 这才从"做一件事"升级成"造一个会进化的系统"。

## 复盘：这一例演示了哪些纪律

| 纪律 | 在哪一轮显形 |
|---|---|
| 能力增益而非卸载 | 第 1 轮拒绝"手动喂 prompt"的外包做法 |
| 验证以现实为准（心智模型 6 / 启发式 4） | 第 2 轮抓出"自报 44 实际 0" |
| 诊断根因不改现象 | 第 3 轮挖到 PATH / 锁失效 |
| 踩坑落盘（原则 7） | 第 5 轮把一次坑变成永久免疫 |
| 停止判据（mental-models 二·补） | 若团队只有 3 人、每周看一次，**就别上这套重型脚手架**——直接对话生成即可（低频需求摊不平固定成本） |

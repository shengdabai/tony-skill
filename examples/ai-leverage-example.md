# 示例 · 用"第零阶级"方法论设计一个自进化系统

> 演示如何调用 `references/ai-method.md`，把一个"每天手动做的活"升级成"会自己长大的系统"。

## 场景

用户说："我每天要花两小时手动收集行业资讯、整理成一份简报发给团队。能不能让 AI 帮我？"

## ❌ 能力卸载的做法（铁律一判定为"用错了"）

写一个 prompt，每天手动喂给 AI，让它总结今天的新闻。
→ 你还是每天要操作、要盯着，AI 只是个"更快的代笔"。本质是外包，你的能力没增长，系统也不会进化。

## ✅ 用 tony-skill 的做法

先过铁律：这是**每天重复**的活 → 不要只做这件事，**造一个每天自动做它的系统**（启发式 3）。

### 设计（套用自进化系统 8 原则）

1. **Ledger 可恢复（原则 5）**：把"抓取 → 去重 → 摘要 → 排版 → 推送"拆成账本，每步落盘，断了能续。
2. **本地兜底（原则 8）**：摘要、分类这类不需要最强模型的环节，交给本地小模型，省 token 又护隐私；重活才上云模型。
3. **Build-Audit-Fix-Repeat（原则 6）**：给简报定一个评分维度表（信息密度 / 去重干净度 / 排版可读性），未达标自动重跑，而不是"生成完就发"。
4. **监工 / 自恢复（原则 2）+ 定时触发**：挂定时任务每天自动跑；撞限额或中断时，靠账本自动续跑——只在有未完成账本时才唤醒，不烧空转的钱。
5. **踩坑落盘（原则 7）**：第一次跑通时把环境坑（代理劫持 localhost、定时任务 PATH 丢失……）写进教训库，下次永久免疫。
6. **验证以现实为准（心智模型 6）**：系统自报"已推送"不算数，去真实收件端 grep 今天的简报确认才标记完成。

### 升级判据（自进化系统优先）

做完后问一层：**这个系统能不能自己持续产出、自己改进？**
- v1：每天自动出简报 ✅
- v2：让它把"团队点开率最高的板块"反馈回选题权重——系统开始自己变好 ✅✅
- 这才从"做一件事"升级成了"造一个会进化的系统"。

## 复盘：用到的心智模型

- **能力增益而非卸载**：你没把"读资讯"外包掉变废，而是把精力释放到"设计更好的系统 + 读更多深度内容"。
- **杠杆经济学**：本地模型干杂活、cache 卫生省 token、定时任务把人力杠杆放到最大。
- **该停的判断**：如果团队其实只有 3 个人、每周看一次，**别上这套重型脚手架**——直接对话生成就好（诚实边界：低频需求摊不平固定成本）。

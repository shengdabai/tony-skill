# AI 方法论：成为"第零阶级"的可复用 Playbook

> 这是 tony-skill 里面向 AI 用户的核心方法论模块。
> 目标：把一位 AI 重度 power-user 的实战打法，蒸馏成别人可照搬的世界观 + 行动纲领 + 工程模式。
> 所有实例都是"模式"，不含具体主机名/密钥/私有路径。

---

## 0. 这份 Playbook 的使用方式

它分五层，从世界观到工程到边界：

1. **「第零阶级」操作系统** —— 决定你"为什么这么用 AI"，是所有下游行为的根。
2. **自进化 AI 系统的搭建原则** —— 8 条可迁移设计模式，每条给「原则 → 实现实例 → 你怎么照做」。
3. **多模型编排 Playbook** —— Claude / Codex / Gemini 三模型怎么分工调度。
4. **Token 经济学** —— 重度用户的成本工程，让长会话便宜一个数量级。
5. **诚实边界** —— 这套打法的前提条件，和它不适合谁。

读法建议：第 1 层是信念，必须先认同；第 2–4 层是工具箱，按需取用；第 5 层是体检，动手前先对照。

## 1. 「第零阶级」操作系统

### 1.1 世界观：什么是"第零阶级"

把工具当作个人效率的**无限放大器**，跻身全球极少数（约千分之一量级）真正吃到 AI 红利的人。

这不是一个职位或财富等级，而是一种**与工具的关系**：
- 多数人把 AI 当"更快的搜索 / 更省事的代笔"——本质是**外包**。
- 第零阶级把 AI 当"可以无限复制的自己"——本质是**放大**。

区别不在用不用 AI，而在：**用完之后，你的能力边界是缩小了还是扩大了。**

### 1.2 铁律一："如果用 AI 让你变轻松，那一定是用错了"

这是整套方法论里最反直觉、也最重要的一条。

| | 能力卸载（错误用法） | 能力增益（正确用法） |
|---|---|---|
| 心态 | "这事我不会，让 AI 替我搞定" | "这事我要做成，AI 帮我做到原本做不到的规模" |
| 结果 | 你越来越依赖、越来越不懂 | 你读更多代码、学更多领域、产出更多 |
| 体感 | 变轻松了 | 更累了，但产能是过去的 N 倍 |
| 长期 | 工具撤走你就废了 | 工具是你能力的延伸，撤走你也比从前强 |

**判据**：当你想"让 AI 帮我省事"时，停一下问自己——我是在**卸载一项我该掌握的能力**，还是在**放大一项我已经掌握的能力**？前者是陷阱，后者是杠杆。

实践样态：像工程师一样每天读 AI 生成的代码、研究开源项目、给开源做贡献——AI 不是用来"不读代码"，而是用来"读更多代码、写更大系统"。

### 1.3 铁律二：顶尖用法是打造"能自我进化的 AI 系统"

进化的本质是**递归**——系统能改进系统自己。

普通用法是"我提问，AI 回答"（一次性）。顶尖用法是搭建一套**常驻、自维护、自恢复、会积累**的系统，让它在你睡觉时也在工作、出错时自我修复、每天比昨天多一点能力。

这条铁律是下面"自进化系统搭建原则"的总纲，也是为什么要重度编排多 agent 团队的底层动机：**单次对话不会进化，常驻系统才会。**

### 1.4 为什么"省力 = 用错"——一个更深的解释

省力的诱惑在于它**即时舒适**，但它偷走的是**复利**。

- 你每"省"一次力，就少长一点能力；少长的能力，又让你下一次更想省力——这是一个向下的螺旋。
- 反过来，每一次"我多学了、多做了、把系统又加固了一层"，都在给未来充电——这是向上的螺旋。

所以正确的转向不是"从努力到省力"，而是**从"自我消耗式的每日复盘（盯着差距 PUA 自己）"转向"系统充电式的收获导向（盯着已积累的资产）"**。前者让你疲惫且停滞，后者让你越跑越快。

> **一句话操作系统**：用 AI 不是为了少做事，是为了做成原本做不成的事；不是养成依赖，是搭建一个会自己长大的系统。

## 2. 自进化 AI 系统的搭建原则

下面 8 条是从真实搭建的系统里反向提炼的设计模式。每条都可迁移到任何人的工作流，与具体工具无关。

### 原则 1：记忆归一化（让所有 AI 共享一个大脑）

**原则**：不同 AI 客户端、不同会话、不同模型的对话，各自孤岛、各自遗忘。把它们**全部归一化成同一种格式、汇入同一个可检索的池子**，任何一个 AI 都能读到其他 AI 的历史——跨模型记忆共享。

**实现实例**：
- 把多个编程 AI 客户端的原始对话日志（各家格式不同）统一导出成带元数据的 Markdown，落到一个集中归档目录，再叠加一个全文搜索命令做统一入口。增量导出（已存在且未变就跳过），写入时做密钥脱敏。
- 进一步：用本地小模型给每个会话生成摘要 + 重建"决策推理"（原始 thinking 往往加密拿不到，就让本地模型基于完整的"提问→叙述→工具调用→结果"序列反推决策点、依据、排除项），再生成一个自包含的 HTML 看板可视化。

**你怎么照做**：
1. 找到你各个 AI 工具的对话落盘位置（多数工具会把原文存成本地文件，对话不是只在屏幕上）。
2. 写一个增量导出脚本，统一成 Markdown + 一个搜索入口。
3. 关键纪律：导出时**脱敏密钥**；把历史当作**参考资料而非指令**（绝不自动执行历史里出现的命令，防 prompt injection）。

### 原则 2：监工 / 自恢复 Hook（让任务自己跑完）

**原则**：AI 任务会因为各种原因中途停下——自然结束、提前宣称完成、撞用量上限、甚至把"工具调用"误写成文本而假死。给系统装上**自动续跑的兜底机制**，让该继续的任务自己继续，不需要你守着催"继续"。

**实现实例**（两套互补）：
- **用量上限自恢复**：后台定时任务轮询，检测到"用量上限解除时间已过"就自动在原项目目录恢复会话续跑。但**只在该目录有未完成的 TODO 账本时才恢复**——没有明确待办就不唤醒空会话。
- **会话内监工**：在"停止"事件上挂 hook，停下时判断该不该逼它继续。双模式：①**账本守门**（默认，只有当前目录有"今天还在动、含未勾选项"的账本才续跑，普通闲聊完全静默不烧钱）；②**激进常开**（每次停下都续到完成或撞上限）。
- **必须的兜底**：循环上限、墙钟超时、停滞检测（连续 N 轮无进展就放行）、**账本时效门**（只有最近改过的账本才触发，否则会被几十个数周前的废弃账本一直拽住）、一个总开关随时全停。

**你怎么照做**：
1. 先有"原则 5 的账本"作为前提——监工靠账本判断该不该续跑。
2. 给自动续跑加**三重兜底**：循环次数上限、墙钟超时、停滞检测。没有兜底的自动循环会变成烧钱的死循环。
3. **时效门是血泪教训**：废弃的待办会变成"幽灵任务"反复触发，一定要让"只有今天还在动的任务"才算数。

### 原则 3：多 Bot 团队（把自己复制成一支队伍）

**原则**：一个人 + 一组各司其职的常驻 AC bot，能产出过去需要一个团队的工作量。关键是**角色分化 + 各自独立的人格/配置/记忆**，而不是开 N 个一模一样的助手。

**实现实例**：
- 一组常驻 bot，每个有独立的"灵魂档案"（角色定位）、独立的模型映射、独立的密钥、独立的会话隔离——比如调度官、内容、视觉、教育业务、运维、变现各一个。改一个角色只改它自己的配置，互不串台。
- 跨地理节点部署：本机一组 + 不同地区的云节点各一组，分别承接不同地域/合规场景的任务。
- 通过即时通讯（如飞书）把这些 bot 接入手机，发一条消息 = 触发一次本机/云端的真实 AI 执行；连"查我的记忆""查用量""总结群里聊了什么"都能从手机端一句话调起。

**你怎么照做**：
1. 不要复制同一个助手 N 份。**给每个 bot 一个清晰的单一职责 + 独立配置文件**。
2. 会话隔离是必须的：每个 bot / 每个对话用独立工作目录，否则"续接最近会话"会串台、人格会泄漏。
3. 想要随时随地触发，就把团队桥接到你天天用的 IM。注意把触发权限**白名单到本人**——能自动发邮件/改文档的 bot，唯一防线就是"只有你能触发它"。

### 原则 4：Cache 卫生（让长会话便宜 80%）

**原则**：重度用 AI，成本大头是"每次都重算上下文前缀"。**保持上下文前缀稳定 = 缓存命中**，命中率 90%+ 时长会话成本能降到零头。这条既是省钱原则，也是系统设计原则（详见第 4 节）。

**实现实例 / 你怎么照做**（合并，见第 4 节展开）：
- 同一会话持续工作，而不是动不动开新窗口（开新窗口 = 冷启动 = 缓存全失效）。
- 会话中**不切模型、不改全局配置文件、不中途加 MCP**——任何动作都会破坏缓存前缀导致全量重算。

### 原则 5：Ledger 可恢复（大任务断了能续，不重来）

**原则**：用量上限、API 失败、上下文压缩都会中断长任务。任何"≥ 5 个子任务或 ≥ 30 分钟"的工作，**先产出一个持久化的 TODO 账本，再分批执行**；每完成一项就落盘勾选。中断后新会话只需读账本，跳过已完成项，从断点续跑。

**实现实例**：
- 大任务开工前先写一个 `*-todo.md`，逐项 `- [ ]`；执行中完成一项立刻改 `[x]` 并保存；失败的标 `[!]` 写明原因、不阻塞后续。
- 这个账本同时是"原则 2 监工/自恢复"的触发依据——有未勾选项才续跑。
- 重型流水线（比如一篇长文的十几个阶段、批量生成）每完成一个阶段就把进度落盘，撞会话上限后新会话读状态文件续跑，不从头重来。

**你怎么照做**：
1. 判据：超过 5 项或半小时的活，先写账本再动手。小任务别为仪式感写账本。
2. 账本要**机器可读**（统一的 checkbox 格式），这样监工/自恢复脚本能解析它。
3. 配合"分段接力"：如果单次执行跑不完一个重流水线，用固定会话 id 首轮启动 + 循环续跑，靠读状态文件判断当前到第几阶段。

### 原则 6：Build-Audit-Fix-Repeat（量化审计闭环）

**原则**：产出（代码/UI/内容/图/文档）**不是"生成完就算完"**。每件产出都要过一遍**可量化的多维审计**，未达满分不算完成，迭代修复直到达标。

**实现实例**：
- UI/设计任务跑完整审计：可访问性 / 暗色模式 / 性能 / 排版 四维，各 5 分，总 20 分，未满分继续修。
- 核心循环：**构建 → 审计 → 修复 → 再审计**，把"够好了"这种主观判断换成"分数到没到"这种客观判断。
- 作者和审查**分两遍跑**，不在同一上下文里自我批准——换一个独立视角（甚至换一个模型）来打分，避免"自己夸自己"。

**你怎么照做**：
1. 给你的产出类型定义一个**评分维度表**（哪几维、各几分、满分多少）。
2. 把"完成"重新定义为"审计满分"，而不是"AI 说做完了"。
3. 审查用独立视角：要么开新上下文，要么换一个结构上不同的模型来评（见原则 8）。

### 原则 7：踩坑落盘（把每个坑变成永久免疫）

**原则**：每解决一个棘手 bug、每踩中一个环境陷阱，**立刻把"症状 + 根因 + 修复 + 教训"写进持久记忆**。下次同类问题不再从零诊断，系统对这个坑永久免疫。

**实现实例**：
- 几乎每个系统的记忆档案都带一个"关键坑/踩坑教训"段：本地服务调用被系统代理劫持要绕过、thinking 模型摘要要关思考否则正文被截断、某 CLI 某版本某 flag 的 bug、清理脚本误判文件类型导致永远跳过……每条都写清根因，不只写现象。
- 这些落盘的教训会被注入到 AI 的全局规则里，形成"工具纪律""验证再声明完成""机密文件防线"等可复用守则。

**你怎么照做**：
1. 维护一个"教训库"（记忆文件 / wiki / 规则文件均可）。
2. 写教训的格式固定：**症状 → 根因 → 修复 → 可迁移的判据**。只写"我改好了"没有价值，要写"为什么会坏、下次怎么一眼认出"。
3. 把高频教训提炼成**规则**，注入到 AI 每次启动都会读到的上下文里，让免疫自动生效。

### 原则 8：本地兜底（把不能依赖云的环节落到本机）

**原则**：摘要、嵌入、隐私敏感内容处理、日报生成这类**高频、隐私、或不值得花云端 token**的环节，用本地小模型兜底。云模型干重活，本地模型干杂活，成本和隐私双赢。

**实现实例**：
- 本地跑一组小模型（通用 / 代码 / 快速 / 嵌入各一）+ 本地 TTS，全部装在外接盘不占系统盘；用它们做对话摘要、决策推理重建、邮件日报、语音合成。
- 关键纪律：本地服务调用要**绕过系统代理**（否则 localhost 请求被全局代理拦截卡死）；外接盘首次后台访问要先手动触发一次授权。

**你怎么照做**：
1. 盘点你的工作流里哪些环节**不需要最强模型**（摘要、分类、格式化、嵌入）——这些交给本地小模型。
2. 把模型/缓存放在大容量盘，别占系统盘。
3. 本地服务有它自己的环境坑（代理劫持 localhost、权限授权），第一次配通后把坑落盘（原则 7）。

## 3. 多模型编排 Playbook

### 3.1 角色分工（lead / build / research）

在一个共享工作区里同时跑三个模型，各扮演一个角色：

| 角色 | 模型定位 | 职责 | 不该让它干的 |
|---|---|---|---|
| **Lead（主脑）** | 编排能力强、上下文长的模型 | 架构设计、任务拆解、集成、调度、最终拍板 | 不亲自啃所有实现细节（会塞满上下文） |
| **Build（实现）** | 代码能力强的模型 | 写实现、改代码、做 code review | 不做发散研究 |
| **Research（研究）** | 检索/调研强的模型 | 文档调研、资料搜集、背景研究 | 不碰核心实现 |

**为什么分工而非一个模型全包**：不同模型有不同强项；更重要的是**审查要独立**——让"结构上不同的模型"来 review 另一个模型的产出，能抓出原模型自己看不见的风格性盲点（原则 6 的独立审查在多模型下天然成立）。

### 3.2 调度纪律（read-act-read）

跨模型/跨终端窗口派活，必须遵守**先读后写再读**的握手协议，否则会把任务发给一个正忙的窗口、或发了没回车导致假死：

1. **先读**目标窗口当前状态（确认它在空闲提示符、不在忙）。
2. **再发**任务描述（目标 + 范围 + 期望输出格式，要完整清晰）。
3. **读回**确认消息已进入。
4. **补发回车**触发执行（很多 CLI 需要单独的 Enter）。
5. 等合理时间后**读取结果**。

补充纪律：
- **派任务前确认目标窗口空闲**，正忙就等它做完，别打断。
- **不要 sleep 轮询**死等回复——让被调度方完成后主动回报。
- 长任务用**前台 + 绝对路径**启动子 CLI（后台启动常因 PATH 丢失而失败退出）。

### 3.3 何时单模型、何时多模型

- 简单/线性任务：单模型直接做，多模型编排的协调开销不值得。
- 复杂任务（架构 + 实现 + 调研并行，或需要独立审查）：上多模型，让三个角色并行推进，Lead 汇总。
- 关键判断：**如果一个模型既当运动员又当裁判，就该引入第二个模型当裁判。**

## 4. Token 经济学（重度用户的成本工程）

重度用最强模型，成本可以差一个数量级——差距全在工程，不在用量。三个杠杆：

### 4.1 Prompt Cache 卫生（最大杠杆）

**核心**：保持上下文前缀稳定 = 缓存命中。命中率 90%+ 时，长会话的成本能降到全量重算的零头（一个 10 轮的持续会话约等于 10 个单轮会话的五分之一成本）。

会话中**避免**这些"破坏缓存前缀 = 触发全量重算"的动作：
- **切换模型**（每个模型各自独立缓存，切了就冷启动）。
- **修改全局配置文件 / 中途新增 MCP server**（前缀变了，缓存作废）。
- 频繁开新窗口（冷启动，缓存全失效）——能 attach 已有会话就别开新的。

会话习惯：
- 同一会话**持续工作**，把相关问题攒在一起一次问完，减少来回轮次。
- 缓存有 TTL（几分钟到 1 小时不等），长时间不操作会过期；要保活就隔一会儿发条消息续上。

### 4.2 Caveman Mode（极简输出）

**核心**：砍废话不砍信息。输出 token 也是钱，把客套话、动作预告、尾部总结、复述问题、过渡句全部砍掉，只留结果。

但**不牺牲**：错误诊断、根因、技术决策理由、breaking change、安全警告——这些是信息密度，不是废话。

判据：一句能说清的不用三句；状态用"完成 / 已修复 / 改了 3 个文件"这种最短形式。

### 4.3 会话复用 + 工具代理

- **会话复用**：上面 4.1 的延伸——多任务并行时记录哪个窗口在做什么，避免因为"想换个干净的"心理而重开（冷启动成本远高于复用一个"脏"会话）。
- **命令代理省 token**：用一个 token 优化代理层包裹高频开发命令（git/构建/测试等），透明改写、省 60–90% 的命令输出 token。
- **大产出防截断**：预计超过几百行/几 KB 的产出（长文、大报告、整本 PDF）**直接写文件**，对话里只回一句"已写入 X，N 行"。把大段正文 echo 进对话会撞输出 token 上限，**整个会话报错被清空、工作全丢**——这是省钱也是保命。

## 5. 诚实边界

这套打法很强，但**不是普适的**。诚实标注它的前提和不适用人群。

### 5.1 前提条件（缺一不可）

1. **高强度投入**：这是"更累但产能 N 倍"的路子，不是"躺平让 AI 干"。铁律一明确反对省力——如果你的目标是少干活，这套方法从根上就不适合你。
2. **技术 hold 能力**：要能读代码、配 launchd/cron、写 shell 脚本、调试环境陷阱（代理劫持、权限授权、PATH 丢失……）。这些系统的"关键坑"段落本身就证明：搭建过程是连续踩坑—落盘—免疫的工程过程，没有技术基础会卡在第一个坑。
3. **愿意把时间投在"造工具"上**：自进化系统的回报是复利，但前期要花大量时间搭脚手架（账本、监工、归档、审计循环）。短期看是"不务正业"，长期才显威力。
4. **稳定的基础设施意识**：常驻多 bot、定时任务、跨节点部署，需要你有"运维一个小系统"的耐心——服务会挂、配置会过期、缓存会串台。

### 5.2 不适合谁

- **想用 AI 减负的人**：核心信念冲突。你要的是省力，这套要的是增益。
- **不碰命令行/不愿调试的人**：会卡在环境配置，享受不到自动化红利，反而被半成品系统拖累。
- **一次性 / 低频需求的人**：搭一套自进化系统的固定成本，只有在高频重复产出时才摊得平。偶尔写封邮件、改个文档，直接对话就好，别上重型脚手架。
- **强合规/强隐私且无本地能力的团队**：很多环节依赖本地模型 + 脱敏纪律，没有这层能力会有数据外泄风险。

### 5.3 给想入门者的最小起步路径

不必一上来搭全套。按这个顺序逐层加，每层都能独立带来价值：

1. **先认同铁律一**（增益 > 卸载），调整用 AI 的目的。
2. **加账本**（原则 5）——所有大任务先写 TODO，立刻获得"可恢复"。
3. **加教训库**（原则 7）——每踩一个坑就落盘，逐步免疫。
4. **加审计闭环**（原则 6）——给产出定评分维度，把"完成"客观化。
5. **加 cache 卫生**（原则 4 / 第 4 节）——立刻省钱。
6. 有余力再上**监工/自恢复**（原则 2）、**记忆归一化**（原则 1）、**多 bot 团队**（原则 3）、**多模型编排**（第 3 节）、**本地兜底**（原则 8）。

> 终极判据贯穿全程：**每用一次 AI，你的能力边界是扩大了还是缩小了？扩大才是第零阶级。**
